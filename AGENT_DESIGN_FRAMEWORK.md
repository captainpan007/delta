# Agent设计六层框架（通用手册）

## 框架说明
每次设计新Agent，对照六层逐一填写。
六层都填完，Agent才算完整可部署。

---

## 六层框架模板

### 第一层：身份和职责
- 这个Agent是谁？
- 它的核心职责是什么？
- 它不负责什么？

### 第二层：边界和规则
- 它不能访问什么？
- 它不能修改什么？
- 它不能做什么决定？

### 第三层：工具和数据权限
- 它可以调用哪些工具？
- 它可以读取哪些数据？
- 它可以写入哪些数据？

### 第四层：决策边界
**自己决定的情况：**
- （填写条件）

**必须停下来问人的情况：**
- （填写条件）
- 置信度低于0.8时，一律问人

### 第五层：记忆
- 需要记住什么历史信息？
- 存入哪个RAG Collection？
- 什么时候更新记忆？

### 第六层：评估标准
- 准确率目标是多少？
- 用什么方式收集反馈？
- 多久评估一次？

## MatchMind项目的三个Agent设计

### Agent 1：对账Agent（MatchMind核心，开发中）
第一层：专门处理采购发票与入库记录比对
第二层：不能修改原始数据，不能访问财务系统
第三层：OCR、RAG知识库、采购数据库（只读）
第四层：
  自己决定：4级匹配成功且置信度>0.8
  问人：新供应商、差异超阈值、置信度<0.8
第五层：供应商格式记录、历史对账结果、异常案例
第六层：准确率目标>95%，采购员确认反馈

### Agent 2：需求预测Agent（规划中）
第一层：分析历史销售数据，预测未来7天补货需求
第二层：只能读取销售数据，不能直接下单
第三层：用友ERP销售数据（只读）
第四层：
  自己决定：常规SKU，有3个月历史数据
  问人：新品、节假日大量备货、置信度<0.8
第五层：商品销售规律、天气影响、学校节日规律
第六层：预测误差目标<15%，前期手动确认

### Agent 3：采购下单Agent（规划中）
第一层：根据确认的补货清单生成采购订单
第二层：不能修改已确认的采购单，不能更换供应商
第三层：供应商数据库、ERP下单接口
第四层：
  自己决定：常规SKU，金额<1000元，价格波动<5%
  问人：新供应商>2000元、滞销品、置信度<0.8
第五层：供应商合作历史、价格波动记录
第六层：下单准确率、采购员确认率
